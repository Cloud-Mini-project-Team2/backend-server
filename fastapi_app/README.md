# FastAPI 인증(JWT) 모듈

## 1. 왜 JWT를 선택했나 — 세션 방식과의 비교

인증 방식은 크게 "서버가 상태를 저장하는 방식(세션)"과 "서버가 아무것도 저장하지 않는 방식(JWT/서명쿠키)"으로 나뉩니다. 둘의 차이와, 이 프로젝트가 JWT를 고른 이유를 정리합니다.

### 세션 방식 (서버가 DB/Redis에 저장)

- 로그인 성공 시 서버가 세션ID를 발급하고, 그 ID를 키로 회원 정보를 DB(또는 Redis)에 저장
- 클라이언트는 세션ID만 쿠키로 들고 다니고, 요청마다 서버가 DB를 조회해서 회원을 식별
- **장점**: 로그아웃/강제 만료 시 서버에서 해당 행만 지우면 그 즉시 완전히 무효화됨. 쿠키가 탈취돼도 그 자체는 무의미한 랜덤 값이라 정보가 노출되지 않음
- **단점**: 요청마다 DB 조회가 한 번씩 추가됨. 트래픽이 아주 많은 서비스에서는 이 조회 자체가 부하가 될 수 있음 (그래서 "세션은 Redis 같은 곳에 두는 게 좋다"는 얘기가 일반적인 모범사례로 통용됨)

### JWT 방식 (서버가 아무것도 저장하지 않음, 이 프로젝트가 선택)

- 로그인 성공 시 회원 정보(`member_id`, `email`, `nickname`)와 만료시간을 담아 서버 비밀키로 서명한 토큰을 발급
- 서버는 요청마다 서명만 검증하면 되고, DB 조회가 필요 없음 (완전히 "상태 없는(stateless)" 방식)
- **장점**: 서버 저장소가 필요 없어 구현이 단순하고, 여러 서버로 확장해도 세션 데이터를 공유할 필요가 없음
- **단점(트레이드오프)**:
  - 로그아웃해도 토큰 자체는 만료시간 전까지 여전히 유효함 (탈취된 토큰은 서버가 즉시 막을 방법이 없음 — 그래서 유효기간을 짧게(1일) 잡음)
  - payload는 서명만 되고 암호화는 안 되어 있어서, 디코딩하면 누구나 내용을 볼 수 있음 (그래서 비밀번호 등 민감정보는 절대 넣지 않음)
  - 서명 비밀키(`JWT_SECRET_KEY`)가 유출되면 그동안 발급된 모든 토큰이 위조 가능해짐 (그래서 `.env`로 분리하고 절대 커밋하지 않음)

### 왜 세션이 아니라 JWT인가

순수 보안 관점에서는 세션(서버 저장) 방식이 더 안전하다고 보는 게 일반적입니다 (즉시 무효화가 가능하기 때문). 다만 이 프로젝트는:

- 라즈베리파이 2B(1GB RAM) 환경이라 Redis 같은 추가 인메모리 저장소를 새로 얹는 부담이 있고
- 소규모 미니 프로젝트라 대규모 트래픽에 따른 DB 부하 문제보다는, 서버 상태를 안 가져도 되는 구현 단순성이 더 중요하다고 판단
- 유효기간을 24시간으로 짧게 잡아서 "토큰 탈취 시 즉시 무효화 불가"라는 단점의 영향 범위를 최소화

라는 이유로 JWT를 선택했습니다. 발표 등에서 "왜 이 방식을 골랐는지" 질문받으면 위 트레이드오프를 그대로 설명하면 됩니다.

## 2. 파일 배치

`fastapi_open_sv.sh`가 만드는 `./fastapi_app/` 폴더 기준으로, 아래 5개 파일을 그 안에 넣으면 됩니다.

```
fastapi_app/
├── main.py          # 기존 샘플 main.py를 이걸로 교체
├── database.py      # 신규
├── security.py       # 신규 (비밀번호 해싱 + JWT 발급/검증)
├── deps.py           # 신규 (인증 확인용 Depends 함수)
└── auth_router.py    # 신규 (signup/login/logout/me 엔드포인트)
```

## 3. 필요 패키지 추가

기존 `requirements.txt`(crawler용과 별개로, FastAPI 앱용)에 아래 3개를 추가합니다.

```
fastapi
uvicorn[standard]
pymysql
python-dotenv
pyjwt
bcrypt
pydantic[email]
```

설치:
```
pip install pyjwt bcrypt "pydantic[email]"
```

## 4. `.env`에 값 추가

`fastapi_app/.env` 파일에 DB 정보와 함께 아래 한 줄을 추가합니다.

```
JWT_SECRET_KEY=본인이_직접_생성한_랜덤값으로_교체
```

**JWT_SECRET_KEY는 본인이 직접 생성**하세요 (이 대화에 노출된 값은 예시일 뿐 실제로 쓰면 안 됩니다 — 채팅 로그에 남은 값은 비밀키로 쓰기에 안전하지 않습니다).

```
python3 -c "import secrets; print(secrets.token_hex(32))"
```

이 명령어로 나온 값을 복사해서 넣으시면 됩니다.

`.env`는 당연히 `.gitignore`에 포함되어 있어야 합니다 (crawler 쪽에서 이미 등록해두셨다면 같은 파일이 아니어도 규칙은 재사용됩니다).

## 5. 실행 및 테스트

```
cd fastapi_app
source venv/bin/activate   # 또는 run.sh로 처음부터 실행
uvicorn main:app --reload
```

브라우저에서 `http://<서버주소>:8000/docs` (Swagger UI) 접속 후 순서대로 테스트:

1. `POST /api/auth/signup` — 회원가입
2. `POST /api/auth/login` — 로그인 (응답 쿠키에 `access_token`이 담기는지 개발자도구 Application 탭에서 확인)
3. `GET /api/users/me` — 로그인한 회원 정보가 정상적으로 나오는지 확인 (쿠키 없이 호출하면 401)
4. `POST /api/auth/logout` — 로그아웃 후 다시 `GET /api/users/me` 호출 시 401 뜨는지 확인

## 6. 구현 요약

- 비밀번호는 `bcrypt`로 해싱해서 저장, 로그인 시 `bcrypt.checkpw`로 비교
- 로그인 성공 시 `member_id`, `email`, `nickname`을 담은 JWT를 발급해서 httpOnly 쿠키(`access_token`)로 전달, 유효기간 24시간 (`security.py`의 `JWT_EXPIRE_HOURS`)
- 알고리즘은 `HS256`(대칭키) 사용 — 서버가 하나뿐인 구조라 비대칭키(RS256)까지는 필요 없다고 판단
- 이메일 존재 여부가 드러나지 않도록, 이메일 없음/비밀번호 틀림을 구분하지 않고 동일한 401 에러로 응답
- `GET /api/users/me`는 실제 마이페이지 API라기보다, 인증 로직이 제대로 동작하는지 확인하기 위한 테스트용 엔드포인트입니다. 실제 마이페이지 응답(희망조건, 최근 지원 공고 등)은 이후 회원 API 작업 시 확장하면 됩니다.
